## 目的

issue-workにおける複数Issueの並列実行メカニズムを定義する。依存関係に基づくWave実行モデルにより、安全性を確保しつつ処理スループットを最大化するための要件である。

## 要件

| ID | 要件 |
|---|---|
| REQ-0030-001 | 複数Issueの依存関係を4レベル（L0: 完全独立、L1: Specs共有、L2: ファイル衝突、L3: 明示的依存）で分類し、LLM意味解析により判定するものとする（SHALL） |
| REQ-0030-002 | 依存関係に基づくWaveスケジューリングを実行し、同一Wave内のIssueは並列実行し、Wave間は直列実行するものとする（SHALL） |
| REQ-0030-003 | 1回の呼び出しで処理可能なIssue数は最大5件とする（SHALL） |
| REQ-0030-004 | 依存関係分析結果はユーザー確認必須とし、ユーザーの承認なく自動実行してはならない（SHALL） |
| REQ-0030-005 | specs更新（system.md・patterns.md）は親エージェントのみが実行し、Issue番号昇順で直列に更新するものとする。サブエージェントによるspecs更新は禁止する（SHALL） |
| REQ-0030-006 | 各Wave内のIssueはサブエージェントで並列実行し、サブエージェントが利用できない場合はSequential Wave（順次実行）にフォールバックしてもよい（MAY） |
| REQ-0030-007 | Wave開始前に親エージェントがEpicステータスを一括更新し、サブエージェントによる同時更新の競合を回避するものとする（SHALL） |

## 適用範囲

- **対象**: 依存関係レベル（L0-L3）の定義、Waveスケジューリング、サブエージェント起動プロトコル、specs直列更新、最大Issue数制限
- **対象外**: 単一Issue実行の内部手順、worktree管理の詳細、PR作成・マージ手順
