## 背景・課題

現行の `issue-work` コマンドは単一 Issue のみ対応。複数の独立した Issue を処理する際、利用者は Issue ごとにコマンドを起動し直す必要がある。独立した Issue（バグ修正のテンプレート削除、設定ファイルの整理等）が溜まった際、1コマンドでバッチ処理できれば手間を大幅に削減できる。

`git-worktree` skill は既に `.worktrees/{N}-{type}` 形式で Issue ごとの worktree 作成をサポートしており、並列実行の基盤は存在する。

## 目標

1コマンドで複数/範囲の GitHub Issue を一括処理できるようにする。Issue 間の依存関係を自動判定し、安全な順序で実行する。単一 Issue 時の現行フローは完全に維持する。

## 機能要件

| ID | 要件 | 受け入れ基準 |
|---|---|---|
| REQ-0017-01 | **多重Issue引数パース**: Step 0 で個別(`1 2 3`)・範囲(`1-5`)・混在(`1 3-5 7`)の引数をパースする。逆順(`5-3`)・同一(`3-3`)・ゼロ・負数はエラー処理する。重複は除去する | Given: 引数 `1 3-5 7` / When: Step 0 パース / Then: Issues #1,#3,#4,#5,#7 に展開。`5-3` → エラー拒否。`3-3` → 単一フォールバック |
| REQ-0017-02 | **Issue妥当性確認**: `gh issue view {N} --json state,title,labels` で各 Issue の存在・状態を確認。存在しない/closed Issue は警告付きスキップ。有効0件時は中止 | Given: 範囲 `1-5` で #3 が存在せず / When: 妥当性確認 / Then: #3 を警告付きスキップ、他は処理継続。全無効 → 中止 |
| REQ-0017-03 | **依存関係分析**: Step 0b で各 Issue の本文・タイトル・ラベルから依存関係を 4レベル(L0-L3) で判定。結果テーブルをユーザーに提示し承認を得る | Given: 3 Issues で #54 が #50 に依存 / When: Step 0b / Then: L3 として検出、Wave 2 に配置。ユーザー承認後に実行開始 |
| REQ-0017-04 | **並列実行オーケストレーション**: 依存関係に基づき Wave を構成。各 Wave 内の Issue はサブエージェントで並列実行（Hybrid モデル）。サブエージェント不可時は Sequential Wave にフォールバック。specs 更新（Step 10）は親エージェントのみが直列実行（Issue番号昇順） | Given: Wave 1 に #74,#46 (L0) / When: 実行 / Then: 2サブエージェント並列起動 → 完了待機 → specs 更新（親のみ直列） |
| REQ-0017-05 | **部分続行ポリシー**: サブエージェント失敗時、該当 Issue をスキップし成功 Issue は継続。全 Wave 失敗時は集約レポートで報告。集約レポートは Issue ごとの状態（✅/❌/⏭️）+ 成功/失敗/スキップ件数を表示 | Given: 3 Issues 中 #54 が失敗 / When: Wave 完了 / Then: #74,#46 は PR 作成、#54 はスキップ。集約レポートに成功2/失敗1を表示 |
| REQ-0017-06 | **後方互換性**: 有効 Issue が 1件の場合、現行 Steps 1-12 を変更なく実行。Steps 1-12 の内容・Phase A/B/C 構成・現行 Guardrails 10項目は一切変更しない | Given: `issue-work 74` / When: 実行 / Then: Step 0 → Steps 1-12 の現行フローと完全同一 |
| REQ-0017-07 | **関連ドキュメント更新**: `issue-guide-phases/SKILL.md` に並列実行パターンの概要追記。`README.md` の issue-work 説明に並列対応を追記 | Given: 改修完了 / When: ファイル確認 / Then: SKILL.md に L0-L3 概要あり、README.md に使用例あり |

## 非機能要件

| 項目 | 要件 |
|---|---|
| Context Window 保護 | 1呼び出しあたり最大 5 Issues（超過時は拒否メッセージで通知） |
| エラー伝播 | 失敗 Issue は兄弟 Issue の実行をブロックしない（部分続行） |
| 拡張性 | 新規 Steps（0b, 0c）は Step 0 の分岐先として追加。既存 Steps は不変更 |
| 実行モデル | Hybrid（親エージェント統括 + サブエージェント並列）を基本。サブエージェント不可時は Sequential Wave にフォールバック |
| specs 整合性 | specs 更新は親エージェントのみ・直列・Issue番号昇順。同一セクション競合時はマージ（既存保持 + 新規追加） |
