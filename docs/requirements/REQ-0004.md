## 目的

要件（REQ）ファイルとアーキテクチャ決定記録（ADR）の管理規則を定義する。REQは領域別の総体として管理し、ADRは取り返しのつかない技術判断を記録する。両者の整合性を維持するための規約である。

## 要件

| ID | 要件 |
|---|---|
| REQ-0004-001 | REQファイルは領域別の総体として管理し、1領域につき1ファイル（`REQ-{NNNN}.md`）とする。要件が肥大化した場合は分割してもよい（SHALL/MAY） |
| REQ-0004-002 | REQファイルの構造は目的・要件・適用範囲を必須セクションとし、Acceptance Criteria・Verification Method・Change Historyセクションを追加してもよい。frontmatterにはid・title・created・updated・tagsのみを含むものとする。status・scaleフィールドは持たない（SHALL/MAY） |
| REQ-0004-003 | REQの要件はRFC 2119言語（SHALL/SHOULD/MAY）で記述し、FR/NFRの区別を持たず単一テーブルに統合するものとする（SHALL） |
| REQ-0004-004 | REQ番号は4桁ゼロ埋め（`REQ-{NNNN}`）で採番し、欠番を再利用してはならない。新規採番は既存最大番号+1とする（SHALL） |
| REQ-0004-005 | REQの操作モデルはCREATE（新規作成）・APPEND（セクション追記）・UPDATE（内容変更）の3種類とし、判定基準に基づいて自動選択するものとする（SHALL） |
| REQ-0004-006 | ADRは取り返しのつかない技術判断のみを記録対象とし、ADR作成の要否は判定基準に基づいて評価するものとする（SHALL） |
| REQ-0004-007 | docs/requirements/README.mdとdocs/README.mdはREQファイルとの整合性を維持し、欠落・不整合を自動検出・修正するものとする（SHALL） |

## 適用範囲

- **対象**: REQファイルの構造・操作モデル・採番規則、ADRの作成基準、REQ/ADRの整合性管理、README インデックス管理
- **対象外**: 要件分析手法（req-analysisスキルの管轄）、仕様適合性検出（spec-complianceスキルの管轄）、docs/specs/ の内容管理
