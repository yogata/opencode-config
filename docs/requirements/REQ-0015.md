## 背景・課題

Issue #82の作成セッション（ses_1f3fd5882ffeez0gx88P27qJcS）で実証されたように、issue/PRの残課題抽出は現在アドホックな手動プロセスである。`gh` CLIでのデータ取得、キーワード検索、分類、Epic+子Issue作成の全工程に手作業と判断が介在し、再実行時に一貫性が担保されない。

## 目標

定期的に（ユーザー主導で）issue/PRの残課題を抽出・分類し、Epic + 子IssueとしてGitHub上に管理するワークフローを、OpenCodeコマンドとして確立する。

## 機能要件

| ID | 要件 |
|---|---|
| REQ-0015-01 | 指定期間内の issue/PR の本文・コメントから残課題を抽出する（2段階抽出戦略） |
| REQ-0015-02 | Phase1: クローズ済みissue/PRの未チェックチェックボックスを構造的検出する |
| REQ-0015-03 | Phase2: LLM全文解析で「対象外」「先送り」「別途対応」等のテキスト的残課題を抽出する |
| REQ-0015-04 | 抽出結果をハイブリッド分類する（基本カテゴリ固定 + 分類判断はLLM） |
| REQ-0015-05 | 分類結果をレポートとしてユーザーに提示し、確認・微調整を経て承認を得る |
| REQ-0015-06 | 承認後、Epic Issue（サマリー + テーブル）を作成する |
| REQ-0015-07 | Epic配下に子Issueを作成し、元Issue/PRへのリンクを付与する |

## 非機能要件

- **実行エージェント**: sisyphus（実行・操作系タスクのため）
- **対象期間指定**: 自然言語による期間指定（「直近1週間」「今月」「2026-05-02から」等）をLLMが解釈
- **冪等性**: 同一期間に対する再実行時に既存Epicの更新でなく新規作成とする（重複防止はユーザー判断）

## スコープ

- **対象**: 残課題抽出 → 分類 → レポート確認 → Epic/子Issue作成のワークフロー
- **対象外**: GitHub Actions等の自動定期実行 / 既存Epicの差分更新 / REQ/ADR/specsの更新
