## 目的

issue系ワークフロー（issue-work / issue-close）において、実装コードだけでなく関連ドキュメントも要件達成対象として扱う。変更後の仕様と既存ドキュメントが矛盾する状態を同一Issue内の未完了事項とし、別Issueに送ることを禁止する。

**背景**: 現状では、実装コードの修正は実施されたが、同じ仕様を説明している `system.md` や `README.md` の更新が別Issue扱いと判断されることがある。しかし、変更後の仕様と既存ドキュメントが矛盾する状態は、同一Issue内の未完了事項である。

## 要件

| ID | 要件 |
|---|---|
| REQ-0015-001 | issue-req の責務は、ユーザーとの壁打ちにより要件・背景・対象外・受け入れ条件を整理することである。関連ドキュメントの個別ファイル列挙をユーザーに求めてはならない（SHALL） |
| REQ-0015-002 | issue-work は、要件に基づき実装コード・設定・スキーマ・cron・環境変数・API・関連ドキュメントを対象に影響範囲を探索するものとする（SHALL） |
| REQ-0015-003 | issue-work は、変更後仕様と矛盾する既存ドキュメントを同一Issue内で更新するものとする。関連ドキュメントがissue-reqに明記されていないことを理由に、更新対象外としてはならない（SHALL） |
| REQ-0015-004 | issue-close は、実装コード・設定・関連ドキュメントが要件と矛盾していないことを確認するものとする（SHALL） |
| REQ-0015-005 | issue-close は、旧仕様の記述が残っている場合、その記述が変更後仕様と矛盾しないことを確認するものとする（SHALL） |
| REQ-0015-006 | 変更後仕様と矛盾するドキュメント更新漏れがある場合、issue-close は完了不可とする（SHALL） |

## 適用範囲

- **対象**:
  - `.opencode/commands/issue/issue-work.md` — 実装時にコード・設定・関連ドキュメントを要件に従って更新する責務を明記
  - `.opencode/commands/issue/issue-close.md` — 完了確認時にコード・設定・関連ドキュメントの整合性を確認する責務を明記
  - issue-work / issue-close が参照する関連スキル — 判断基準・完了報告・品質ゲートに関わる記述の更新
  - フェーズ責務・完了条件・SSoT遷移に関わる横断スキル — `issue-lifecycle` 等の横断的定義の反映
- **対象外**:
  - 新規ツールの実装
  - 新規スクリプトの実装
  - ユーザーに関連ドキュメント探索を求める運用への変更
  - issue-req に関連ドキュメント列挙責務を持たせる変更
  - issue-reqで全関連ファイルを列挙させる運用への変更
  - 仕様変更と直接関係しないドキュメント再構成・表現改善の強制
