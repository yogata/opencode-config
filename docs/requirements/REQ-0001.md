## 目的

Issue駆動開発ワークフローの全体構造を定義し、フェーズ間の移行条件・SSoT遷移・エラー回復の規則を明確にする。ワークフロー全体の一貫性を担保するための基盤要件である。

## 要件

| ID | 要件 |
|---|---|
| REQ-0001-001 | ワークフローは3つのマクロフェーズ（バイブス壁打ち・構造的実行・レビュー完了）で構成され、各マクロフェーズは1つ以上のマイクロフェーズを含むものとする（SHALL） |
| REQ-0001-002 | 各マクロフェーズにおけるSingle Source of Truth（SSoT）を定義し、フェーズ境界を跨ぐ情報の唯一性を保証するものとする（SHALL） |
| REQ-0001-003 | マクロフェーズ①から②への境界において、docs変更（REQ・ADR・README等）をコミット・プッシュしなければならない。これにより②フェーズのworktreeがdocs変更を継承する（SHALL） |
| REQ-0001-004 | バグ修正（Pattern A）、機能追加（Pattern B）、refactor/maintenance（Pattern C）、docs/chore（Pattern D）の4つのワークフローパターンを定義し、ラベル体系に基づいて自動判定するものとする（SHALL） |
| REQ-0001-005 | Pattern Bにおいて規模判定基準（複数モジュール跨ぎ・PR肥大化リスク・段階的リリース）のいずれかを満たす場合、Epic規模として扱うものとする（SHALL） |
| REQ-0001-006 | 各コマンドはフェーズ境界ルールに従い、自身の入力SSoTと出力SSoTを明示的に定義するものとする（SHALL） |
| REQ-0001-007 | ワークフロー実行中のエラー発生時、ユーザーに判断を委ねるか、安全なフォールバック動作に切り替えるものとする。データ損失を伴う自動回復（ユーザーの未コミット変更を破棄する操作を含む）は実行してはならない（SHALL） |
| REQ-0001-008 | 各Patternのlabel→Pattern mappingを `bug`→A, `enhancement`/`feature`→B, `refactor`/`maintenance`→C, `docs`/`chore`→D と定義するものとする（SHALL） |
| REQ-0001-009 | Pattern A〜Dの対応表（Type labels, Branch type, REQ要否, ADR要否, specs更新要否, Review focus）を定義するものとする（SHALL） |
| REQ-0001-010 | label と metadata/template-type が矛盾する場合、コマンドは停止してユーザー確認を求めなければならない。自動推定で矛盾を解消してはならない（SHALL） |
| REQ-0001-011 | マクロフェーズ①のSSoTはIssue作成状態に応じて遷移する。Issue未作成時は壁打ちセッション内の要件doc（draft）、Issue作成後はIssue本文とする。コマンドは両状態を扱えるものとする（SHALL） |

## 適用範囲

- **対象**: フェーズ体系定義、SSoT遷移ルール、フェーズ境界ルール、Pattern A/B/C/D判定基準、label→Pattern mapping定義、Pattern対応表、label矛盾検知ルール、規模判定基準、エラー回復方針
- **対象外**: 各コマンドの内部手順、並列実行の詳細、REQ/ADRファイルの管理操作、スキルの品質基準
