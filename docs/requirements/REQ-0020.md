## 背景・課題

`issue-backlog` で作成される Epic Issue には、子Issueのステータスを追跡するテーブル（☐ 未着手 / 🔄 進行中 / ✅ 完了）が含まれる。しかし、このテーブルは初期値（☐ 未着手）のまま更新されず、`issue-close` での「✅ 完了」更新まで変化がない。「🔄 進行中」ステータスを設定するコマンドが存在しないため、進行中の作業状況が Epic 上で可視化されない。

## 目標

- `issue-work` が子Issueの作業を開始した際、親 Epic Issue のステータス追跡テーブルを「🔄 進行中」に自動更新する
- 単一Issueモード・多重Issueモードの両方で正常動作する
- `issue-close` との整合性を保つ（完了時の「✅ 完了」更新は既存実装のまま）

## 機能要件

| ID | 要件 |
|---|---|
| REQ-0020-01 | `issue-work` Phase A（準備完了）時、子Issue本文から `Parent: #{N}` パターンを検出し、親Epic Issueのステータス追跡テーブルの該当行を `☐ 未着手` → `🔄 進行中` に更新する |
| REQ-0020-02 | 親Epicが存在しない場合（`Parent:` パターンなし）、ステータス更新をスキップしエラーとしない |
| REQ-0020-03 | 多重Issueモードの場合、親エージェントが各Wave開始前に、該当Wave内の全子IssueのEpicステータスを一括更新する（サブエージェントによる同時更新の競合を回避） |
| REQ-0020-04 | ステータス値は既存の3値（`☐ 未着手` / `🔄 進行中` / `✅ 完了`）に従い、新しいステータス値を追加しない |
| REQ-0020-05 | `issue-close` Step 8 の既存Epic更新処理（`✅ 完了` への更新・自動クローズ）は変更しない |
| REQ-0020-06 | `gh-cli-best-practices` に従い、Epic本文更新は `--body-file` で実行する |

## 非機能要件

- **べき等性**: 既に `🔄 進行中` の場合、重複更新せずスキップする
- **安全性**: Epic本文の他セクション（サマリー・Completion Criteria等）を破損しないこと。正規表現による対象行のピンポイント置換を行う
- **フォールバック**: Epic本文の取得・更新に失敗した場合、警告を表示して `issue-work` 自体は継続する（ブロックしない）

## スコープ

- **対象**: 
  - 新規スキル `epic-status-tracker` の作成（ステータス追跡プロトコルの知識ベース）
  - `issue-work.md` の変更（新スキルのload、Phase Aへの更新ステップ追加）
  - `issue-close.md` の変更（新スキルのload、Step 8の知識をスキル参照に統合）
- **対象外**: 
  - `issue-backlog` テンプレート（`issue_desc_epic.md`）の変更
  - `issue_desc_child.md` テンプレートの変更
  - 新しいステータス値の追加（レビュー中等）
  - GitHub Project等の外部ツール連携
