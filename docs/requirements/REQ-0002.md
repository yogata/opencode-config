## 目的

8つのissue-*コマンドが従うべき入出力契約・実行順序・データフローを定義する。コマンド間の結合度を明示し、ワークフロー全体の整合性と拡張性を担保するための通信規約である。

## 要件

| ID | 要件 |
|---|---|
| REQ-0002-001 | issue-reqはセッション会話を入力とし、要件docを出力するものとする。出力には既存REQとの照合結果を含め、UPDATE対象項目とCREATE対象項目を分類して明記するものとする。照合プロセスの詳細はREQ-0002-009〜011に従う。対話系コマンドとしてprometheusエージェントを使用する（SHALL） |
| REQ-0002-002 | issue-save-reqは`.sisyphus/drafts/`のドラフトを入力とし、REQファイル・ADRファイル・インデックス更新を出力するものとする。draft-metaの指定に従い、複数のREQファイルのUPDATE・CREATEを一括で実行するものとする（SHALL） |
| REQ-0002-003 | issue-createは要件docおよびspecs・ADRを読み取り、GitHub Issueを出力するものとする。Epic規模の場合はEpic + 子Issueを一括作成する（SHALL） |
| REQ-0002-004 | issue-workはGitHub Issueおよびspecsを読み取り、実装済みブランチ・PRを出力するものとする。3フェーズ構成（準備・実装・提出）でべき等な再開ポイントを提供する（SHALL） |
| REQ-0002-005 | issue-closeはPRとIssueを入力とし、マージ完了・記録追記・ブランチ削除を実行するものとする。REQステータス更新および学び抽出を含む（SHALL） |
| REQ-0002-006 | issue-nextは現在のセッション状態を分析し、次に実行すべきコマンドを推論して提示するものとする（SHALL） |
| REQ-0002-007 | issue-backlogはユーザー指定の期間内にクローズされたIssue/PRから残課題を抽出し、解消済み確認を実施した上で backlog draft（`.sisyphus/drafts/backlog-draft-{period-slug}.md`）を作成するものとする（SHALL） |
| REQ-0002-023 | issue-backlog-createは承認済み backlog draft を入力として Epic + 子Issueを作成し、抽出元 issue/PR に `backlog-extracted` コメントを投稿するものとする（SHALL） |
| REQ-0002-008 | issue-updateはIssue本文の更新・コメント追加に加え、REQファイルのAPPEND/UPDATE操作をサポートするものとする（SHALL） |
| REQ-0002-009 | issue-reqは壁打ちの意図把握後（Step 1完了時）、要件展開（Step 2）の前に、`docs/requirements/REQ-*.md` をスキャンし、ユーザーの要件説明と既存REQ（タイトル・タグ・目的・要件内容）の関連性を推論して該当REQを特定するものとする（SHALL） |
| REQ-0002-010 | 照合で関連REQが特定された場合、issue-reqはその内容を壁打ちコンテキストに即時反映し、要件の具体化に活用するものとする。関連REQの目的・要件・適用範囲を提示し、ユーザーとともに変更点を深掘りする（SHALL） |
| REQ-0002-011 | issue-reqは照合結果に基づきreq-operation（CREATE/APPEND/UPDATE）およびtarget-reqを確定し、draft-metaの対応フィールドに記録するものとする。複数REQが該当する場合は、それぞれに対する操作を個別に指定するものとする（SHALL） |
| REQ-0002-012 | issue-reqは引数なしで単体実行された場合、Step 0（セッションコンテキスト検知）を最初に実行するものとする。引数ありの場合はStep 1から開始する（SHALL） |
| REQ-0002-013 | Step 0は現在のセッションの会話履歴を分析し、以下の6項目を依存関係順に推論するものとする（SHALL）: ①要件内容（何をやりたいか）→ ②Pattern判定（A or B）→ ③Scale判定（Pattern Bのみ、Aでは推論不要）→ ④ADR判断（必要/不要）→ ⑤要件doc構造化（テーブル形式要件の有無）→ ⑥適用範囲（対象/対象外） |
| REQ-0002-014 | Step 0は各推論項目に信頼度（高/低）を付与するものとする。高信頼度は「セッション内で明示的に言及・合意されている」場合、低信頼度は「暗黙的に推論可能だが明示的な言及がない」場合とする（SHALL） |
| REQ-0002-015 | Step 0は`.sisyphus/drafts/req-draft-*.md`の存在を確認し、ファイル名のtopic-slugとセッションの要件内容の一致を検証するものとする。トピック不一致の場合はdraftを無視し、セッションコンテキストを優先する（SHALL） |
| REQ-0002-016 | Step 0はルーティング前に推論サマリーをユーザーに表示するものとする。サマリーは各項目の推論結果と信頼度を陈述形式（質問形式ではない）で提示する（SHALL） |
| REQ-0002-017 | ユーザーが推論サマリーに対して「違う」「やり直し」と応じた場合、Step 0はStep 1から通常壁打ちを開始するものとする（SHALL） |
| REQ-0002-018 | Step 0のルーティング判定は以下の分岐に従うものとする（SHALL）: **全項目高信頼度かつdraft存在** → Step 9（要件doc確認）へスキップ、**全項目高信頼度かつdraftなし** → Pattern Bの場合Step 7（ドラフト保存）、Pattern Aの場合Step 9へ、**一部項目が低信頼度または未推論** → 推論済み項目を継承し不足項目のみ対象にStep 1（壁打ち）を開始、**推論結果なし** → 通常Step 1から開始 |
| REQ-0002-019 | Step 0はセッション内で複数の異なる機能/バグが議論されている場合、直近のコンテキスト（最後の要件議論）を使用するものとする。ただし話題が混在して推論が困難な場合はStep 1にフォールバックする（SHALL） |
| REQ-0002-020 | Step 0の全ルーティング経路はStep 9（要件doc確認）を通過するものとする。これにより要件docの最終提示と差し戻し判定を担保する（SHALL） |
| REQ-0002-021 | Step 0は既存Steps 1-9のロジックを一切変更しないものとする。Step 0は既存Stepの直前に挿入される独立したステップである（SHALL） |
| REQ-0002-022 | Step 0は過去セッションの読み取りを行わないものとする。現在のセッションコンテキストのみを推論ソースとする（SHALL） |

## 適用範囲

- **対象**: 9つのissue-*コマンドの入出力契約、実行順序、コマンド間データフロー、エージェント指定、issue-reqの既存REQ照合プロセス、issue-reqのセッションコンテキスト検知（Step 0）
- **対象外**: コマンド内の詳細手順、並列実行の依存分析、テンプレートの内容、gh CLIの安全性手順、過去セッションの読み取り
