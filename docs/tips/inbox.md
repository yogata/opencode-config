## 2026-05-10: issue-backlog由来Issueの不明確さへの対処方針

- **事象**: issue-backlogで抽出した子Issue（例: staff-schedule#739）が、元issue/PRの「対象外」1行文のみを出典とするため、概要がタイトルの再述に過ぎず極めて不明確になった
- **原因**: issue-backlogは①バイブス壁打ちフェーズのショートカット経路であり、構造的抽出のみを行う設計。元ソースに情報が少ない場合、深掘り能力がない（設計上の制約）
- **対策**: 方針C — backlog Issueは「まだらな未整理リスト（プレースホルダー）」として許容し、着手時にissue-reqで壁打ち深掘りする。フロー: issue-backlog（粗いIssue作成）→ 着手時にissue-req（壁打ち）→ issue-save-req（docs保存）→ issue-update（既存Issue更新）→ issue-work（実装）。既存Issue更新はissue-createではなくissue-updateを使用する点に注意。
- **タグ**: #ワークフロー #issue-backlog #要件定義 #プロセス設計

## ADR管理スキルの設計パターン

**状況**: adr-file-manager スキルの新規作成と adr-guidelines の更新を行った
**原因**: ADR管理機能が整理されておらず、ガイドラインとファイル管理ロジックが混在していた
**解決策**: 責務分離 — adr-guidelines（評価基準・判定）と adr-file-manager（採番・CRUD・バリデーション）に分離
**教訓**: スキル設計では「判定ロジック」と「管理ロジック」を分離すると保守性が向上する。同一セクション構造を採用することで一貫性のある運用が可能。ADR のステータス値（proposed/accepted/deprecated/superseded-by）は REQ と異なる体系を持つ。
**タグ**: #スキル設計 #ADR #責務分離

## Markdown コマンド定義の段階的拡張

**状況**: issue-work コマンドに複数Issue並列実行を追加（135行→244行）
**原因**: 既存の単一Issue フローを壊さずに新機能を組み込む必要があった
**解決策**: Step 0 の条件分岐で新旧パスを分離。分岐点で「有効数 = 1 → 従来フロー、≥2 → 新モード」とし、Steps 1-12 の内容は一切変更しない。新モード専用の Step 0b/0c/Phase D を追加。
**教訓**: コマンド定義の後方互換性を保つ拡張パターン — 既存 Step の内容不変 + 分岐点でのみ新規パスを注入。単一ケース時のオーバーヘッドをゼロに保てる。
**タグ**: #コマンド設計 #後方互換性 #段階的拡張

## LLM + 正規表現ハイブリッド分析

**状況**: 複数Issue間の依存関係判定を設計
**原因**: 正規表現単独では「トリプル化に伴い」等の暗黙的依存を見逃す（バックテストで判明）
**解決策**: LLM意味解析を主、正規表現を補助とする2層構成。4レベル分類（L0-L3）の明確な枠組みをプロンプトに与えて判定品質を安定化。
**教訓**: 自然言語に埋め込まれた暗黙的関係は正規表現では検出不可。LLM意味解析を主軸に据え、正規表現は明示的パターン（#N 等）の高速検出に限定する構成が実用的。
**タグ**: #依存関係 #ハイブリッド分析 #プロンプト設計

## 並列PRマージ時のbase変更コンフリクトパターン

**状況**: issue-work で Wave 2（#85,#86,#87）を並列実装し、3つのPRを順次マージした
**原因**: 先にマージされたPRがmain（base）を更新するため、後続PRがコンフリクトになる。特に#86と#85が共にissue-work.md Step 8を修正していたため同一箇所でコンフリクト
**解決策**: worktree上で `git rebase origin/main` → コンフリクト解消 → `git push --force-with-lease` → 再マージ
**教訓**: 並列Issueで同じファイルを修正する場合、マージ順序を意識して最も変更の大きいPRを最後にするか、コンフリクト解消手順を事前に準備する
**タグ**: #並列実行 #コンフリクト #issue-work

## worktree使用中ブランチのローカル削除制約

**状況**: `gh pr merge --delete-branch` でリモートブランチは削除されたがローカルブランチ削除が失敗
**原因**: worktreeがそのブランチを参照している間は `git branch -d` が失敗する
**解決策**: 先に `git worktree remove` でworktreeを削除してから `git branch -d` を実行
**教訓**: `--delete-branch` オプションはリモートのみ削除。worktree使用中のローカルブランチはworktree削除後に手動削除が必要
**タグ**: #git-worktree #ブランチ管理

## PR Closes #NN による自動Issueクローズ

**状況**: 全4PRに `Closes #NN` を含めていたため、マージ直後にIssueが自動クローズされた
**原因**: GitHubの自動クローズ機能（PR本文の `Closes`/`Fixes`/`Resolves` キーワード）
**解決策**: 手動 `gh issue close` が不要だった（"already closed" エラーで確認）
**教訓**: PR本文に `Closes #NN` を含めれば、マージ時の自動クローズを活用できる。issue-close では自動クローズを前提とし、手動クローズはフォールバックにする
**タグ**: #GitHub #自動化 #issue-close

## テンプレート準拠の3層防御パターン

**状況**: PR #129がテンプレート（`pr_desc.md`）の構造を無視した出力（`Summary/Root Cause/Changes`）を生成
**原因**: コマンド定義は「テンプレートに従って生成」と指示するのみで、(1)テンプレートに必須/任意の区別がなく、(2)コマンドに必須セクション一覧がなく、(3)検証ゲートがなかった
**解決策**: 3層防御 — (1)テンプレート側に `<!-- 【必須】 -->` / `<!-- 【任意】 -->` マーカーを付与、(2)コマンド側の該当Stepに必須セクション一覧を明記、(3)各コマンドのGuardrailsに準拠ルールを追加
**教訓**: 「テンプレートに従って生成」という指示だけでは、エージェントはテンプレート構造を無視する可能性がある。参照（READ）→必須セクション明示（SPECIFY）→検証ゲート（VERIFY）の3層で初めて準拠が担保される。これはコマンド設計の一般原則として適用可能。
**タグ**: #テンプレート準拠 #コマンド設計 #検証ゲート #品質担保
