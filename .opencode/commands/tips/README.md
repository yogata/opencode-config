# tips コマンド群

学びの3層パイプライン（キャプチャ → 分析 → 昇華）を提供するコマンド・スキル群。
基本方針は「入口を緩く、出口を厳しく」。

## 構成

3つのコマンド + 1つの補助スキルで構成される。

| コマンド/スキル | 役割 | 層 |
|---|---|---|
| `tips-capture`（スキル） | エージェント主体で学びを検知・抽出・提案 | キャプチャ層 |
| `/tips/tips-add` | 確定済み学びをinbox.mdに保存 | キャプチャ層 |
| `/tips/tips-refactor` | 問題クラス分類 → 8軸評価 → archive移動 | 分析層 |
| `/tips/tips-elevate` | 昇華判定 → stagingスタブ生成 | 昇華層 |

## 3層アーキテクチャ

```
┌─────────────────────────────────────────────────────────────┐
│ キャプチャ層                                                │
│  tips-capture (skill): エージェント主体で学びを検知・抽出    │
│  /tips/tips-add:       確定済み学びを inbox.md に保存       │
└──────────────────────┬──────────────────────────────────────┘
                       │ inbox.md
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ 分析層                                                      │
│  /tips/tips-refactor: 問題クラス分類・8軸評価・archive移動  │
│  出力: archive.md + evaluation-report.md                    │
└──────────────────────┬──────────────────────────────────────┘
                       │ evaluation-report.md + archive.md
                       ▼
┌─────────────────────────────────────────────────────────────┐
│ 昇華層                                                     │
│  /tips/tips-elevate: 廃棄判定・スタブ生成・prune            │
│  出力: elevation-staging/                                   │
└─────────────────────────────────────────────────────────────┘
```

## データフロー

```
inbox.md → (tips-refactor) → archive.md + evaluation-report.md → (tips-elevate) → elevation-staging/
```

反映ルート（stagingスタブから実装へ）:

```
elevation-staging/ → issue-req → issue-save-req → issue-create → issue-work
```

## データファイル

| ファイル | 場所 | 役割 |
|---|---|---|
| `inbox.md` | `docs/tips/inbox.md` | 未処理の学び。`tips-add` / `tips-capture` で追加される |
| `archive.md` | `docs/tips/archive.md` | 生きている tips プール。未処分・保留中・再評価対象を保持。prune・elevate時pruneで動的に変化する |
| `evaluation-report.md` | `docs/tips/evaluation-report.md` | 評価済み中間レポート。**毎回上書き、追記運用しない**。tips-refactor と tips-elevate 間の境界成果物 |
| `elevation-staging/` | `docs/tips/elevation-staging/` | 中間スタブ置き場。**正式成果物ではなく、issue-req経由で実装に移行するための一時領域** |

## コマンド・スキル詳細

### tips-capture（スキル）

エージェント主体で学びを検知・抽出・提案するスキル。13フィールド形式でエントリを生成し、ユーザーの承認/却下を求める。

**トリガー条件**（以下のタイミングで検討）:

- `/issue/issue-close` の実行中
- バグを修正したとき
- 調査が完了し、原因が判明したとき
- CI/CDパイプラインが失敗したとき
- テンプレートや仕様からの逸脱を修正したとき
- gh/git操作でワークアラウンドが必要だったとき
- 実装エラーによりIssue更新が必要になったとき
- エラーの解決策を見つけたとき

**基本原則**: エージェントが学びを検知・抽出・提案し、ユーザーは承認または却下のみを行う。

**入口を緩く原則**: false positiveを恐れず、品質評価は下流（refactor/elevate）に委ねる。重複の統合もrefactorが行う。

### /tips/tips-add

確定済み学びを `docs/tips/inbox.md` に追記するコマンド。

- **ダムセーブ専用**: 情報の収集・昇華判定・ADR/REQ/spec反映判断は行わない
- 13フィールド形式のエントリを受け取って永続化する
- `inbox.md` が15件以上の場合、`/tips-refactor` の実行を提案する

**引数**:

- `$1` — タイトル（必須）
- `$2` — タグ（オプション、カンマ区切り）
- コンテンツ — 13フィールド形式の本文

### /tips/tips-refactor

`inbox.md` と `archive.md` の学びエントリをセマンティック分析し、`evaluation-report.md` を出力後、inbox から archive へ移動するコマンド。

**10ステップの処理フロー**:

1. **inbox.md の内容確認** — ファイルの存在確認とエントリ数カウント
2. **archive.md の読込** — 存在すれば読込、なければ空として扱う
3. **全エントリ読込 + 旧フォーマット正規化** — 旧5フィールド形式を新13フィールド形式にマッピング（元ファイルは書き換えない）
4. **問題クラス分類** — 根本原因 + 再発条件 + 予防策が同じエントリを同一クラスにグループ化。**クラスタリングは本文ベース、タグは補助情報**
5. **8軸評価スコアリング** — 各クラスに対し8軸（発生件数/影響度/横展開性/反映先明確度/自動化適性/プロジェクト固有知識再利用性/再発可能性/費用対効果）で1〜5スケール評価
6. **evaluation-report.md 生成** — **毎回上書き、追記しない**
7. **refactor時prune（任意）** — archive.md内の古い単発レアケースを削除候補として特定。**必須ではない**
8. **ユーザーに分析結果提示** — アーカイブ移動の承認を求める
9. **アーカイブ移動（原子的操作）** — inbox → archive へ移動。archive書込成功確認後にinboxをクリア
10. **完了報告 + elevate提案** — 問題クラスが検出されていれば `/tips-elevate` を推奨

**prune対象の特定基準**（全てを満たすもの）:

- 記録から一定期間（目安: 3ヶ月以上）経過している
- その後、本質的に同じ問題が再発していない
- 影響度が低い（スコア2以下）
- 再発条件が曖昧
- 横展開性が低い（スコア2以下）
- 費用対効果が低い（スコア2以下）

**削除禁止エントリ**（以下のいずれかを含む場合はprune不可）:

- 判断基準を含む tips
- 技術知識を含む tips
- プロジェクト固有知識を含む tips

### /tips/tips-elevate

`evaluation-report.md` の問題クラスを主入力とし、廃棄判定を行って `elevation-staging/` にスタブを生成するコマンド。

**処理フロー**:

1. evaluation-report.md の存在確認
2. evaluation-report.md + archive.md の読込
3. **廃棄判定**（11カテゴリ + duplicate）

**廃棄カテゴリ一覧**:

| # | カテゴリ | 判定基準 |
|---|---|---|
| 1 | 既存 command へ反映 | 既存コマンドに追加すべき手順・制約 |
| 2 | 既存 skill へ反映 | 既存スキルに追加すべき知見 |
| 3 | 新規 skill 化 | 汎用パターン、独立した判断・手順が確立 |
| 4 | 新規 command 化 | 繰り返し現れる操作フロー、自動化すべき手順が明確 |
| 5 | template 反映 | テンプレート形式に反映すべきフォーマット知見 |
| 6 | ADR 候補 | アーキテクチャに関する設計判断・技術選定の理由 |
| 7 | spec 候補 | システム仕様・実装パターンとして docs/specs/ に反映すべき内容 |
| 8 | REQ 候補 | 要件変更・機能追加の要因となる知見 |
| 9 | project-local knowledge | プロジェクト固有の落とし穴・環境依存の知見 |
| 10 | deferred | 昇華の余地がない、情報が断片的、出現回数が少ない |
| 11 | rejected | ユーザーが明示的に却下、すでに別の対策で十分対応済み |
| + | duplicate | 既存の command/skill/template/docs で同等内容が十分カバー済み |

**既存対策確認**: 既存コマンド/skill/template/docsに類似対策が存在するか確認し、「新規X化」より「既存Xへ反映」を優先する。

**既存コマンド・Skill に同種の責務がある場合、既存反映を優先する。**

**スタブ確認観点**: 既存重複確認、反映先、粒度。

**prune**: staged（スタブ生成済み）/ rejected / duplicate のエントリのみ archive.md から除去。deferred / 未処理は保持。

## エントリ形式（13フィールド）

各学びエントリは以下の13項目で構成される。

| # | フィールド | 必須度 | 説明 |
|---|---|---|---|
| 1 | 問題事象 | 必須 | 何が起きたか。事実ベースで客観的に |
| 2 | 発生局面 | 必須 | どの局面で発生したか（実装/CI/レビュー/デプロイ/運用） |
| 3 | 検知方法 | 必須 | どうやって気づいたか |
| 4 | 根本原因 | 必須 | なぜ起きたか。表面的でなく根本的に |
| 5 | 自律対応内容 | 必須 | エージェントが行った具体的アクション |
| 6 | ユーザー確認有無 | 必須 | あり / なし の二値 |
| 7 | ADR/REQ/spec影響 | 必須 | 「なし」でも明記。ユーザー確認「あり」の時は特に注意 |
| 8 | 横展開観点 | 推奨 | 他の場所への展開可能性 |
| 9 | 再発条件 | 推奨 | どのような条件で再発するか |
| 10 | 予防策候補 | 推奨 | 再発防止のアイデア |
| 11 | 想定反映先 | 推奨 | どこに反映すべきか |
| 12 | 関連 | 任意 | ファイルパス、Issue番号等 |
| 13 | タグ | 任意 | `#タグ` 形式、スペース区切り |

**日付形式**: `YYYY-MM-DD`

**エントリフォーマット**:

```markdown
## YYYY-MM-DD: タイトル

- **問題事象**: ...
- **発生局面**: ...
- **検知方法**: ...
- **根本原因**: ...
- **自律対応内容**: ...
- **ユーザー確認有無**: ...
- **ADR/REQ/spec影響**: ...
- **横展開観点**: ...
- **再発条件**: ...
- **予防策候補**: ...
- **想定反映先**: ...
- **関連**: `file/path`
- **タグ**: #xxx #yyy

---
```

## 承認ポイント

tipsワークフローには3つのユーザー承認ポイントがある。

### 1. capture → add 間

エージェントが抽出した学びの内容を提示し、ユーザーが承認または却下する。

- エージェントが13フィールド形式のエントリを生成して提示
- ユーザーは「はい / いいえ」で応答
- 却下された場合は追加しない

### 2. refactor → archive移動間

tips-refactor が分析結果を提示し、ユーザーがアーカイブ移動を承認する。

- evaluation-report.md の内容（問題クラス一覧、8軸評価スコア、全体傾向）を表示
- ユーザーが承認しない場合、アーカイブ移動をキャンセル（evaluation-report.mdは保存済み）
- prune候補がある場合は、pruneの承認も個別に求める

### 3. elevate → staging間

tips-elevate が昇華判定結果を提示し、ユーザーが判定を確認・修正した上で承認する。

- 廃棄判定テーブル（クラスタ / テーマ / 廃棄判定 / 既存対策 / 理由）を提示
- ユーザーが判定の変更を指示可能
- 承認したクラスタのみスタブ生成
- prune計画（staged/rejected/duplicateの除去）も別途承認

## 禁止事項

| 禁止事項 | 該当コマンド/スキル | 理由 |
|---|---|---|
| ユーザーに学びの有無を問わない | tips-capture | 「学びはありましたか？」等の問いかけは禁止。エージェントが自ら判断する |
| 自動的に学びを追加しない | tips-capture | ユーザーの承認なしに inbox.md へ追記しない |
| ADR/REQ/spec反映判断を行わない | tips-capture | 影響の記録まで。実際のADR/REQ/spec更新は別スキルの責務 |
| 昇格判断を行わない | tips-capture | スキルやコマンドへの昇格判断は `/tips-elevate` の責務 |
| `.opencode/` への直接配置禁止 | tips-elevate | スタブは `elevation-staging/` のみに生成。実装は `issue-req` ルートで移行する |
| `elevation-ledger.md` は生成しない | tips-elevate | 管理用ファイルは作成しない |
