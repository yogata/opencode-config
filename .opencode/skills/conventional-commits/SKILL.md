# conventional-commits

## Overview
Conventional Commits v1.0.0に準拠したコミットメッセージの作成を支援します。Angularコミュニティで確立された11種類の標準的なコミットタイプをサポートし、日本語でのコミットメッセージ作成も可能です。 SemVerとの連携により、変更の重大度を自動で判断できるコミットメッセージを生成します。

## When to Use

**Conventional Commitsを使用すべき状況：**
- プロジェクトのコミット履歴を一貫性のある形式で管理したい
- SemVer（Semantic Versioning）に基づいたリリース管理を行っている
- チームメンバー間でコミットメッセージの形式を統一したい
- 変更の種類をすぐに識別できるコミット履歴が必要
- CHANGELOGの自動生成ツールを使用している
- 日本語でコミットメッセージを作成したい

**使用すべきでない状況：**
- 個人的な小規模プロジェクトで形式が気にならない場合
- 既存のプロジェクトで異なるコミット規約が確立されている場合
- 短期間で破棄される一時的な作業の場合

## Quick Reference

| コミットタイプ | 説明 | SemVerレベル | 日本語例 |
|--------------|------|-------------|----------|
| `feat` | 新機能の追加 | MINOR | `feat: ユーザー認証機能を追加` |
| `fix` | バグ修正 | PATCH | `fix(api): レスポンスのステータスコードを修正` |
| `docs` | ドキュメントのみの変更 | PATCH | `docs: READMEのインストール手順を更新` |
| `style` | コードスタイルの変更 | PATCH | `style: コードのインデントを修正` |
| `refactor` | リファクタリング | PATCH | `refactor: 関数のネストを整理` |
| `perf` | パフォーマンス改善 | PATCH | `perf: クエリの実行速度を向上` |
| `test` | テストの追加・修正 | PATCH | `test: ログインのユニットテストを追加` |
| `build` | ビルドシステム変更 | PATCH | `build: Node.jsバージョンを16から18に更新` |
| `ci` | CI/CD設定変更 | PATCH | `ci: GitHub Actionsのバージョンを更新` |
| `chore` | その他の変更 | PATCH | `chore: 依存パッケージを更新` |
| `revert` | コミットの取り消し | PATCH | `revert: feat: ユーザー認証機能を追加` |

| 形式 | 説明 | 例 |
|------|------|-----|
| `<type>` | 必須：コミットタイプ | `feat` |
| `<scope>` | 任意：変更の範囲 | `api`, `auth`, `ui` |
| `!` | 任意：破壊的変更 | `feat!` または `feat(api)!: ` |
| `<description>` | 必須：変更内容の簡潔な説明 | `ユーザーログイン機能を追加` |

| フッター | 説明 | 例 |
|---------|------|-----|
| `BREAKING CHANGE` | 破壊的変更の詳細 | `BREAKING CHANGE: 認証APIのエンドポイントが変更されました` |
| `Refs` | 参照先 | `Refs: #123` |
| `Closes` | 閉じる | `Closes: #456` |

## Implementation

### ステップ1: 基本的なコミットメッセージの作成

```
<type>[optional scope]: <description>
```

**シンプルな例：**

```
feat: ユーザー登録機能を追加
fix: ログイン時のバグを修正
docs: APIドキュメントを更新
```

**スコープ付きの例：**

```
feat(auth): ユーザー認証機能を追加
fix(api): レスポンスのステータスコードを修正
style(components): ボタンのスタイルを調整
perf(database): クエリのパフォーマンスを改善
```

### ステップ2: 破壊的変更を示す

**方法1: `!`構文を使用**

```
feat!: データベーススキーマを変更
feat(api)!: 認証エンドポイントを完全に変更
fix!: 重大なバグを修正
```

**方法2: フッターを使用**

```
feat: ユーザーデータの構造を変更

BREAKING CHANGE: usersテーブルにemailカラムが必須になりました
```

### ステップ3: 詳細な説明を追加する

```
feat: 複数ファイルのアップロード機能を追加

ユーザーが一度に最大10個のファイルをアップロードできるようになりました。
対応するファイル形式: PDF, DOCX, JPG, PNG

Closes: #789
```

### ステップ4: スコープのベストプラクティス

**推奨されるスコープ例：**

```
feat(auth): 認証関連の新機能
feat(api): APIエンドポイントの変更
feat(ui): ユーザーインターフェースの改善
feat(database): データベース関連の変更
feat(config): 設定ファイルの変更
feat(test): テスト関連の新機能
```

**スコープなしの例（プロジェクト全体の変更）：**

```
feat: プロジェクト全体の新しい機能
fix: プロジェクト全体のバグ修正
refactor: 大規模なリファクタリング
```

### ステップ5: 日本語コミットメッセージのベストプラクティス

**良い例：**

```
feat: ユーザー認証機能を追加
- JWTを使用した認証を実装
- ログイン/ログアウト機能を追加
- パスワードリセット機能を実装

fix(api): ユーザー登録時のバグを修正
- メールアドレスのバリデーションを改善
- 重複登録を防止するチェックを追加

docs: APIドキュメントを更新
- 認証APIのドキュメントを追加
- リクエスト/レスポンスの例を更新

perf: ユーザーリストの表示速度を改善
- データベースクエリを最適化
- キャッシュを実装

BREAKING CHANGE: 認証方式が変更されました
- 従来のセッションベースからJWTベースに変更
- 既存のクライアントは対応が必要です
```

**避けるべき例：**

```
add user authentication (英混在)
add feature for login (英混在)
update files (具体的でない)
bug fix (具体的でない)
work on auth (作業内容になっていない)
```

### ステップ6: 複数の変更を1コミットにまとめる

関連する複数の変更を1コミットにまとめる場合：

```
feat: 記事投稿機能を追加

- 記事作成フォームを実装
- マークダウンエディタを追加
- 画像アップロード機能を実装
- 記事の公開/下書き機能を追加

Closes: #123, #456
Refs: feature/article-posting
```

## Common Mistakes

| 間違い | 影響 | 修正 |
|----------|--------|------|
| 大文字と小文字の混同 | ツールで正しく認識されない | 全て小文字で統一: `Feat:` → `feat:` |
| スコープの省略 | 変更範囲が不明確になる | 適切なスコープを指定: `fix:` → `fix(api): ` |
| 説明文が具体的すぎる | コミット履歴が冗長になる | 簡潔に: 「ユーザー認証機能を追加」 |
| 説明文が抽象的すぎる | 変更内容が理解できない | 具体的に: 「認証関連の修正」 → 「ログイン時のバグを修正」 |
| 英語と日本語の混在 | 一貫性が失われる | 全て日本語または全て英語で統一 |
| 破壊的変更を示さない | 重大な変更を誤って適用 | `!`または`BREAKING CHANGE`フッターを使用 |
| ピリオドを末尾につける | 不要な文字が含まれる | ピリオドを省略: 「機能を追加。」 → 「機能を追加」 |
| スペースや特殊文字を含む | システムで正しく処理されない | アンダースコアやハイフンを使用 |

## Real-World Impact

- **コミット履歴の可読性向上**: 30秒で変更の種類と範囲が理解可能
- **SemVer自動判定**: `feat`→MINOR, `fix`→PATCH の自動判断により、バージョン管理が簡素化
- **CHANGELOG自動生成**: コミットメッセージから自動で変更履歴ドキュメントを生成可能
- **チーム開発効率**: 新メンバーでもコミット規約を即座に理解できる
- **リリース計画の明確化**: 変更の重大度を視覚的に把握可能
- **コードレビューの効率化**: 変更の種類でレビューフォームをカスタマイズ可能
- **ドキュメンテーションの一貫性**: 技術文書とコミット履歴の整合性が保たれる
- **デプロイ自動化**: コミットタイプに基づいたデプロイ戦略の実装が可能
