# ADR（Architecture Decision Record）評価ガイドライン

このスキルは、ユーザーのリクエストや提案された変更がアーキテクチャ的に重要であるかどうかを評価し、ADRを作成すべきかどうかを判断するために使用します。

## 評価基準

以下の3つの基準のいずれかに該当する場合、ADRの作成を推奨してください。

1. **アーキテクチャ上の重要性 (Architecturally Significant)**
   - システム全体の構造、主要なコンポーネント間の関係に影響を与えるか？
   - 技術スタックの選択（プログラミング言語、フレームワーク、ミドルウェアなど）に関わるか？
   - 共通のデータモデル、認証・認可スキーム、セキュリティ設計に影響するか？
   - システム全体にわたる関心事（ロギング、エラーハンドリング、通信プロトコルなど）に関わるか？

2. **長期的影響 (Long-term Impact)**
   - この決定が将来の開発や運用に長期的に影響を与えるか？
   - メンテナンスコストや技術的負債に大きく寄与する可能性があるか？

3. **逆転の困難さ (Hard to Reverse)**
   - この決定を後で変更する場合、コストが非常に高い、あるいは困難であるか？

## 判断チェックリスト

- [ ] 新しい主要なライブラリやフレームワークを導入しようとしているか？
- [ ] 既存のディレクトリ構造やパッケージ構成を大幅に変更しようとしているか？
- [ ] データベースのスキーマ設計やストレージ戦略を変更しようとしているか？
- [ ] 外部サービスとの連携方式（API、認証など）を決定・変更しようとしているか？
- [ ] プロジェクト全体のコーディング規約や設計原則を定義・変更しようとしているか？

## 具体例

### ✅ ADR作成が推奨されるケース

- PostgreSQLからMongoDBへの切り替え
- 認証システム（Better Auth, Supabase Authなど）の選定と実装方針
- モノリスからマイクロサービスへの移行、またはその逆
- REST APIからGraphQLへのアーキテクチャ変更
- ディレクトリ構成の根本的な見直し

### ❌ ADRが不要なケース

- ボタンの追加や文言の変更
- 既存のアーキテクチャに基づいた単一機能（フィーチャー）の実装
- バグ修正
- CSSによるスタイルの微調整
- ドキュメントの誤字脱字修正

## 出力形式

評価の結果、以下のいずれかの形式で結論を出力してください。

- **ADR作成が必要な場合:**
  ADR作成推奨: [ここにアーキテクチャ的な重要性や判断理由を記述]

- **ADR作成が不要な場合:**
  ADR不要: [ここに不要と判断した理由（既存アーキテクチャ内での変更であることなど）を記述]
